# Пример: Принцип "Tell, Don't Ask" (TDA)

## Описание

Данный пример демонстрирует принцип **"Tell, Don't Ask" (TDA)** - объекты должны выполнять действия, а не раскрывать свое состояние для принятия решений другими объектами.

## Суть примера

В классе `MyCollection` показаны два подхода к добавлению элементов:

### ❌ Плохой подход (Ask, Don't Tell)
```cpp
void Add(PairGenerator& pg) {
    vector.push_back(pg.createPair());  // Просим генератор создать объект
}
```

### ✅ Хороший подход (Tell, Don't Ask)
```cpp
void Add(MyPair &&other) {
    vector.push_back(other);  // Передаем готовый объект
}
```

## Проблема "Ask, Don't Tell"

Когда `MyCollection` вызывает `pg.createPair()`, она:
- **Нарушает инкапсуляцию** - берет на себя ответственность за создание объекта
- **Увеличивает связанность** - зависит от внутренней логики `PairGenerator`
- **Усложняет код** - смешивает логику создания и хранения

## Преимущества "Tell, Don't Ask"

- **Инкапсуляция** - каждый объект отвечает за свою область ответственности
- **Меньшая связанность** - `MyCollection` не знает, как создаются объекты
- **Гибкость** - можно передавать объекты из разных источников

## Принцип

Вместо того чтобы спрашивать объект о его состоянии и принимать решение за него, лучше сказать объекту, что делать, и позволить ему самому принять решение.
